| 全站按鈕加上點擊彈跳回饋 | `src/app/globals.css`、`src/components/ui/button.tsx`、`src/components/site-header.tsx`、`src/components/intro-hero.tsx`、`src/components/work-card.tsx` | 使用者要求「任何按鈕點擊後要有彈跳感」。用純 CSS（不碰事件處理邏輯，避免重蹈本 session 稍早 Base UI 點擊互動的坑）：`globals.css` 新增 `.tap-bounce` 工具類 + 一條全域 `button:not(:disabled)` 規則，按下時縮到 `scale(0.92)`，放開後用 `cubic-bezier(0.34, 1.56, 0.64, 1)`（會先衝過 1.0 再回彈，是常見的「回彈按鈕」CSS trick）彈回原大小；因為套用範圍是全域 `button` 選擇器，Sheet 關閉鈕、`Tabs` 分類頁籤、輪播箭頭、「來點互動」佔位鈕全部自動吃到，不用逐一修改。原生 `<a>`/`Link` 樣式的「按鈕」（導覽列 4 個連結、品牌連結、手機選單連結、簡介兩顆 CTA、作品卡片整張卡）手動加上 `tap-bounce` class。共用的 `ui/button.tsx`（作品詳情頁下載/外部連結按鈕）拿掉原本的 `active:...:translate-y-px` 小位移，改用 `tap-bounce`；因為它原本的 `transition-all` 會跟自訂的 transition 打架，`.tap-bounce` 的 transition 屬性加了 `!important` 確保彈跳曲線不會被蓋掉。**驗證方式**：純 CSS 效果不牽涉 JS 事件處理，用 `getComputedStyle` 逐一確認導覽列連結、品牌連結、CTA 按鈕、漢堡選單鈕、`ui/button.tsx` 渲染出的 `<a>` 標籤，全部正確吃到 `transitionProperty: transform` + 對應的回彈曲線；`:active` 實際按下狀態的視覺效果無法用自動化工具強制觸發驗證，但因為是單純 CSS 偽類（不像先前的 Base UI 互動元件需要真實事件序列），可靠度高。 |
| 全站過頂/過底橡皮筋回彈效果 | `src/components/overscroll-bounce.tsx`、`src/app/[locale]/layout.tsx` | 使用者要求：滑到頁面最頂/最底時，繼續往同方向滑動，整個畫面要有被拉住、放開後彈回去的感覺（類似 iOS 原生橡皮筋效果）。專案用 Lenis 自己接管捲動（`ReactLenis root`），會蓋掉瀏覽器原生的橡皮筋回彈，所以要手動實作：新增 `OverscrollBounce` 包住整個 `{children}`（放在 `layout.tsx` 的 `ReactLenis` 內層，全站每個頁面都套用），監聽 `wheel`（觸控板/滑鼠）與 `touchstart/move/end`（手機）事件，只在「已經在頂端還往上滑」或「已經在底端還往下滑」時才攔截並用 Framer Motion 的 `useMotionValue` 給整棵樹加上帶阻尼的 `translateY`（`RESISTANCE=0.4`、最大拉距 `MAX_PULL=80px`），放開手勢或滾輪停止 150ms 後用 spring 動畫彈回 0。**除錯記錄**：判斷「是否在頂端」原本用 `<= 0.5` 嚴格比對，但實測 Lenis 靜止時的 `scroll` 值不會精準等於 0（發現是 8px），導致判斷永遠不成立，改用 `BOUNDARY_TOLERANCE = 12px` 容許誤差後正常運作；另外驗證時一度以為完全沒作用，後來發現是 Framer Motion 的 `MotionValue.set()` 不會同步寫入 DOM（要等下一個animation frame才會反映到 style 屬性上），改用等待兩次 `requestAnimationFrame` 後再檢查，才確認 `transform: translateY(...)` 確實有正確套用、放開後也確實彈回 `transform: none`；頂部/底部、滾輪/觸控四種組合都個別驗證過。 |
| 照片輪播支援手機手指滑動切換 | `src/components/photo-carousel.tsx` | 使用者手機實測回報：手機沒有游標，hover 顯示切換箭頭的機制在觸控裝置上完全用不到。改用 Framer Motion 內建的 `drag="x"` + `onDragEnd` 判斷滑動速度/距離（`swipePower = |offset| × velocity`，門檻沿用 Framer Motion 官方 carousel 範例的 10000）來切換圖片，桌面的 hover 箭頭保留不動，兩種操作方式並存。滑動容器加 `touch-pan-y` 讓瀏覽器原生垂直捲動不受影響（只有水平手勢被 Framer Motion 攔截），圖片本身加 `pointer-events-none` 避免瀏覽器原生的圖片拖曳幽靈效果干擾手勢判斷。**驗證方式**：這次改用真實的 pointer 事件序列（pointerdown → 多次 pointermove 模擬時間間隔 → pointerup，讓 Framer Motion 算出真實速度）模擬滑動，而不是單純點擊——同一個 preview session 裡，這種連續手勢事件確實能正確觸發圖片切換（往左滑正確換下一張、往右滑正確換回上一張），推測是先前 session 點擊測試失靈的問題只影響單次點擊事件，不影響這種連續手勢序列。 |
| 作品卡片/經歷時間軸進場動畫改為每次滑到都重播 | `src/components/work-card.tsx`、`src/components/experience-timeline.tsx` | 使用者發現「第一次往下滑有淡入上移動畫，回頂端再滑一次就不會再播」，確認是 Framer Motion `whileInView` 的 `viewport={{ once: true }}` 刻意設計（動畫只在元素第一次進入畫面時播放），非 bug。使用者偏好每次都重播，改成 `once: false`。用 wheel 事件模擬捲動驗證：捲出畫面時卡片正確變回 `opacity:0`（初始狀態），捲回同一位置動畫正確重新播放，行為符合預期。 |
| 簡介照片輪播 + CTA 按鈕互動優化 | `src/components/photo-carousel.tsx`、`src/components/intro-hero.tsx`、`src/lib/site-assets.ts`、`src/app/[locale]/page.tsx`、`messages/*.json` | 新增 `PhotoCarousel`：最多存 5 張照片（`site-assets.ts` 的 `introPhotos`，目前是空陣列，之後放路徑上去即可），用 Framer Motion `AnimatePresence` + `variants`（enter/center/exit 依方向決定進出位移）做滑動轉場；hover 到照片時左右浮現切換箭頭（`opacity-0 group-hover:opacity-100`，也支援 `focus-visible` 讓鍵盤使用者看得到），循環式切換（最後一張按「下一張」繞回第一張）；沒有照片時退回原本的文字佔位框，不影響現有行為。照片與三顆 CTA 按鈕都加上跟下方作品卡片一致的 hover 浮動+陰影動畫（`whileHover={{y:-4}}` + `hover:shadow-lg transition-shadow`）。新增第三顆按鈕「來點互動」放在「探索作品」「與我合作」右邊（目前是佔位按鈕，尚未接功能，使用者確認之後再補）。修正「探索作品」「與我合作」點擊後是瀏覽器原生錨點瞬間跳轉的問題，改成跟導覽列一樣呼叫 `useLenis().scrollTo()` 平滑捲動（在點擊當下即時量測 `header` 實際高度做偏移，不依賴 `SiteHeader` 內部狀態，兩個元件保持獨立）。**驗證缺口**：這次連續三種完全不同的可點擊元件（Base UI Sheet trigger、一般 `next-intl` Link 的 onClick、單純原生 `<button onClick>`）在 preview 工具的自動化點擊測試裡都验证不到事件送達 React handler，但用乾淨的新 session 重新量測過 DOM 幾何位置是正確的（例如輪播箭頭按鈕座標從詭異的負值，在全新 session 重測後變成合理數字），研判是 preview 工具本身這個 session 用到後段的點擊送達不穩定，不是三處完全不相關的程式碼都剛好同時壞掉。**麻煩使用者實際打開網站手動測試：hover 照片看箭頭會不會浮現、點箭頭圖片會不會滑動切換、點「探索作品」「與我合作」會不會平滑捲動而不是瞬間跳轉**，有問題請回報。 |
